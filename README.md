# RetailCast — Retail Demand Forecasting and Decision Intelligence Platform

End-to-end machine learning system for weekly retail demand forecasting, inventory decision support, and LLM-generated business recommendations. Built to demonstrate applied ML engineering, MLOps discipline, and production-oriented software design on real retail data at scale.

| Dataset | Validation | Champion Model | Engineering |
|---|---|---|---|
| 421,570 weekly sales records | 5-fold walk-forward CV | SARIMA — 3.95% MAPE | FastAPI · MLflow · Docker · 83 tests |

---

## Project Overview

RetailCast is a full-stack forecasting platform that connects statistical and machine learning models to downstream supply chain and business decisions. The system ingests Walmart weekly sales data across 45 stores and 81 departments, trains and compares multiple forecasting approaches under time-series-safe validation, selects a champion model, packages it for serving, and exposes predictions through a REST API and interactive dashboard.

The platform extends beyond prediction. Forecast outputs drive inventory optimization (Safety Stock, Reorder Point, Economic Order Quantity), stockout and overstock risk classification, data drift monitoring, and structured business recommendations generated by a hosted large language model with output validation.

This is not a notebook-only exercise. The repository includes modular Python packages, automated experiment execution, model artifacts, API serving, containerization, CI validation, and a test suite covering forecasting, serving, inventory logic, drift detection, and LLM integration.

---

## Business Problem

Retail demand volatility directly affects revenue, inventory carrying cost, and customer availability. Forecast error creates two costly failure modes:

- **Under-forecasting** leads to stockouts, lost sales, and emergency replenishment.
- **Over-forecasting** leads to excess inventory, markdown pressure, and working capital tied up in slow-moving stock.

Organizations need forecasting systems that are accurate under temporal constraints, comparable across model families, reproducible in training, and deployable as a service that downstream planning systems can consume.

RetailCast addresses this by producing validated weekly demand forecasts, ranking models on held-out future windows, promoting a champion artifact into production serving, and translating forecast signals into inventory and operational recommendations.

## Key Achievements

- Processed **421,570** transactional weekly sales records spanning **45 stores**, **81 departments**, and **143 weeks** of history.
- Evaluated **SARIMA**, **Prophet**, and **XGBoost** under **5-fold walk-forward cross-validation** with a 12-week forecast horizon.
- Selected **SARIMA** as champion at **3.95% MAPE**, outperforming Prophet (5.20%) and XGBoost (6.73%) on company-level forecasting.
- Automated champion retraining, serialization, and metadata packaging for reproducible deployment.
- Deployed inference through **FastAPI** with health checks, lazy model loading, and endpoint-level usage metrics.
- Connected forecasts to **inventory optimization** and **risk classification** for operational decision support.
- Integrated **Hugging Face Inference Providers** (Llama 3.1 8B) for structured retail recommendations with validation guards.
- Delivered **83 automated tests** across **19 test modules** and a **GitHub Actions** pipeline covering imports, unit tests, and Docker image build.

```
   │  /monitoring/*      │      │                       │     │                     │
   └──────────┬──────────┘      └───────────────────────┘     └─────────────────────┘
              │
   ┌──────────▼──────────┐
```

**Design principle:** separate data preparation, experimentation, champion promotion, serving, and decision intelligence into distinct modules with explicit interfaces. This keeps training logic independent from inference, allows artifact-based deployment, and supports incremental extension without rewriting the pipeline.

## Core Capabilities

### Forecasting Pipeline

| Component | Implementation |
|---|---|
| Data ingestion | `WalmartDataLoader` with schema validation and deduplication |
| Aggregation | Company, store, and store-department weekly sales series |
| Model families | SARIMA (pmdarima auto-tuning), Prophet, XGBoost |
| Feature engineering | Lag features, rolling statistics, calendar features for tree-based models |
| Validation | Expanding-window walk-forward CV via `TimeSeriesSplit` — no random shuffling |
| Model selection | Champion chosen by lowest mean MAPE across folds |
| Retraining | Champion retrained on full available history before packaging |
| Artifacts | Joblib model serialization with JSON metadata and CSV leaderboard |

**Why walk-forward validation:** random train/test splits leak future information in time series. Walk-forward evaluation trains only on past data and tests on the next horizon, approximating how the model would be used in production.

### Inventory Optimization

Forecast outputs feed a deterministic inventory engine that computes:

| Output | Purpose |
|---|---|
| **Safety Stock (SS)** | Buffer against demand variability at a target service level |
| **Reorder Point (ROP)** | Inventory threshold that triggers replenishment |
| **Economic Order Quantity (EOQ)** | Cost-balanced order size given holding and ordering costs |

A separate **risk classification** module evaluates current inventory position against forecasted demand and returns stockout and overstock risk levels (Low / Medium / High / Critical). This closes the loop between statistical forecast and actionable supply chain signal.

### GenAI Decision Intelligence

The platform integrates a hosted LLM layer that converts forecast metrics into structured business recommendations.

| Aspect | Detail |
|---|---|
| Provider | Hugging Face Inference Providers (OpenAI-compatible router) |
| Primary model | `meta-llama/Llama-3.1-8B-Instruct` |
| Input | Store, department, horizon, historical/forecast sales, trend direction |
| Output format | Three markdown sections: demand analysis, inventory recommendations, operational actions |
| Reliability | Response validation (length, domain relevance); deterministic rule-based fallback on API or validation failure |

The LLM does not replace the forecast model. It interprets model outputs for business stakeholders — a pattern common in production analytics systems where deterministic models produce numbers and language models produce narrative context.

## Model Performance Results

Company-level forecasting results from 5-fold walk-forward validation:

| Rank | Model | RMSE | MAE | MAPE |
|---:|---|---:|---:|---:|
| 1 | **SARIMA** | $2,700,491 | $1,937,834 | **3.95%** |
| 2 | Prophet | $3,676,451 | $2,612,769 | 5.20% |
| 3 | XGBoost | $4,985,476 | $3,433,547 | 6.73% |

**Champion configuration:** `SARIMA(1,0,0)(0,0,2,52)` — first-order autoregressive structure with second-order seasonal moving average on a 52-week cycle.

**Selection rationale:** SARIMA captured the dominant seasonal structure in aggregate weekly retail sales more effectively than Prophet and XGBoost on this dataset and evaluation protocol. Prophet remained a strong interpretable baseline. XGBoost, despite richer feature engineering, did not outperform on company-level series under walk-forward evaluation.

The champion is retrained on the full training set, serialized to `artifacts/models/champion_model.pkl`, and loaded by `ForecastService` at API startup.

## Engineering Highlights

**Temporal integrity in evaluation.** Walk-forward cross-validation prevents lookahead bias. Model comparison is based on out-of-sample future windows, not in-sample fit.

**Separation of training and serving.** The champion pipeline produces versioned artifacts. The API loads artifacts independently, supporting deployment without re-running training code.

**Graceful degradation in serving.** If the champion artifact is missing or incompatible at startup, the API still launches and returns explicit errors on inference routes rather than crashing the process.

**Typed API contracts.** All serving endpoints use Pydantic schemas, reducing integration errors for downstream consumers.

**Testable decision logic.** Inventory optimization and risk classification are pure functions with dedicated unit tests, independent of model inference.

**Controlled LLM integration.** The LLM client validates generated text before returning it to callers. A deterministic fallback ensures the system remains functional when the inference provider is unavailable.

**Reproducible operations.** Docker packages the API layer. CI validates imports, runs the full test suite, and confirms the image builds on every push.

## Quick Start

```bash
# Setup
python -m venv .venv
source .venv/bin/activate          # Windows: .\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
cp .env.example .env               # Add HF_API_TOKEN for live LLM insights

# Train models and generate champion artifact
python run_experiments.py

# Start API (terminal 1)
python run_api.py

# Start dashboard (terminal 2)
streamlit run streamlit_app.py
```

**Docker (API):**

```bash
docker compose up --build
```

**Verification:**

```bash
python verify_all.py
pytest
```
